## 문제의식

Article의 출발점은 software AI와 robotics 사이의 비대칭이다. Language app을 만들 때는 보통 foundation model API를 호출하고, application-specific prompt나 fine-tuning만 고민한다. 반면 robot system을 만들 때는 controller, data pipeline, model training, robot interface, deployment runtime을 거의 모두 직접 만들어야 한다.

PI가 말하는 `physical intelligence layer`는 이 gap을 줄이기 위한 abstraction이다. 단일 checkpoint만 뜻하는 것이 아니라, general-purpose VLA model을 robot application 위에 연결하고, partner-specific data와 runtime loop로 성능을 끌어올리는 stack 전체를 가리킨다.

| software analogy | robotics에서 대응되는 문제 | layer가 줄이려는 비용 |
| --- | --- | --- |
| LLM API | reusable physical intelligence model | task마다 low-level policy를 처음부터 훈련하는 비용 |
| prompt / fine-tune app | robot-specific instruction, data, action interface | domain-specific behavior adaptation 비용 |
| model provider infrastructure | data collection, policy runtime, evaluation, human intervention loop | 배포 중 생기는 long-tail failure 처리 비용 |
| app ecosystem | homes, hospitals, offices, warehouses, factories의 robot use case | use case 탐색과 morphology 전환 비용 |

## Layer로 읽는 핵심 주장

공식 글은 `π0`, `π0.5`, `π0.6`, `π*0.6` 같은 general-purpose robotic foundation model이 robotics application 개발의 하부 layer가 될 수 있다고 주장한다. 이때 layer는 다음 구성요소를 함께 포함한다.

| 구성요소 | article에서 보이는 역할 | 공개/비공개 경계 |
| --- | --- | --- |
| foundation model | π0, π0.5, π0.6, π*0.6 계열 모델이 robot task policy의 출발점이 된다. | π0/π0.5/OpenPI 일부는 공개, partner deployment의 최신 stack은 비공개에 가깝다. |
| partner-specific data | Weave는 WPT, Ultra는 UPT처럼 partner data를 pre-training에 포함한 조건을 비교한다. | data 자체, collection protocol, filtering detail은 공개 article만으로는 재현 불가다. |
| supervised fine-tuning | chart label은 SFT 조건을 명시한다. | exact dataset size, optimizer, train steps, checkpoint는 article에서 공개하지 않는다. |
| robot interface | partner가 own hardware를 PI model에 interface한다고 설명한다. | observation/action schema, controller, safety layer는 공개하지 않는다. |
| deployment runtime | live laundromat/customer site, live warehouse/customer orders에서 실행된다. | production monitoring, intervention UI, rollback/safety protocol은 공개하지 않는다. |
| human-in-the-loop | Ultra는 model stumble 시 human intervention system이 correctness를 보장하고 새 data를 만든다고 설명한다. | intervention trigger, labeling format, learning loop detail은 공개하지 않는다. |

이 글을 읽을 때 가장 중요한 점은 “모델이 좋다”보다 “모델, partner data, SFT, runtime, intervention feedback을 묶어 layer처럼 공급하려 한다”는 방향성이다.

## Partner Evidence

Article은 두 partner section을 통해 research demo가 아니라 real-world setting에서 model을 stress-test한다는 논리를 만든다.

| partner | domain | 공식 글에서 강조한 setting | 핵심 evidence |
| --- | --- | --- | --- |
| Weave | laundry folding, laundromat/customer setting | San Francisco real laundromat/customer site, home robotics로 확장하려는 laundry folding | π0.6 autonomy 증가, Weave data pre-training으로 missed grasp와 intervention 감소 |
| Ultra | e-commerce order packaging, customer warehouse setting | live customer warehouses, real customer orders, industrial workstation에 slot-in하는 robots | π0 -> π0.5 -> π0.6 generation마다 intelligence/throughput/reliability 개선, π0.6 full-shift 96.4% autonomy video caption |

이 evidence는 논문식 독립 benchmark가 아니라 partner-authored deployment report다. 따라서 “실제 배포에서 이런 신호가 있었다”는 근거로는 중요하지만, baseline/control protocol이 완전히 공개된 실험 결과처럼 읽으면 안 된다.

## Weave: Laundry Folding Deployment

Weave section은 laundry folding을 home robotics의 첫 task로 본다. 이유는 universal, time-consuming, attention-intensive이고, household size에 따라 workload가 반복적으로 커지기 때문이다. 동시에 deformable garment manipulation은 long-horizon이며, corner misplacement가 final fold quality에 영향을 주고, garment size/fabric/color/shape variation이 크다.

Article이 공식적으로 적은 Weave evidence는 다음과 같다.

| 항목 | 공식 글 기준 내용 | 해석 |
| --- | --- | --- |
| deployment context | Weave robots의 live deployment videos는 San Francisco real laundromat/customer site에서 나온 것이라고 설명한다. | lab tabletop demo보다 domain noise가 크다. |
| task variety | t-shirts뿐 아니라 long-sleeves, shorts, pants 등 다양한 garment를 접어야 한다고 설명한다. | deformable long-tail이 task의 핵심이다. |
| continuous video | π0.6가 customer site에서 laundry folding하는 continuous shot을 제공한다. | cherry-picked single success clip보다 강한 deployment signal이지만, 전체 실패율을 대체하지는 않는다. |
| metric 1 | Autonomy / total time: chart상 π0.5 SFT보다 π0.6 SFT가 크게 높다. | model generation upgrade가 autonomy에 기여했다는 주장이다. |
| metric 2 | Weave data included in pre-training, 즉 +WPT가 missed grasp sequences를 42% 줄였다고 caption이 말한다. | partner data가 pre-training에 들어갈 때 domain-specific manipulation failure가 줄었다는 주장이다. |
| metric 3 | +WPT가 interventions per full laundry load를 50% 줄였다고 caption이 말한다. | remote specialist 부담 감소가 deployment metric으로 쓰인다. |

여기서 `WPT`는 Weave data included in pre-training이다. `SFT`는 supervised fine-tuning이다. 중요한 구조는 “π0.6 SFT만으로도 π0.5 SFT보다 autonomy가 오른다”와 “π0.6 안에서도 Weave data를 pre-training에 넣으면 missed grasp와 intervention이 줄어든다”를 분리해 읽는 것이다.

Weave section의 tricky situation videos는 fabric top layer grasp, tangled pants legs unraveling, sweater sleeve/collar recovery, fold 아래 edge picking 같은 장면을 보여준다. 이는 benchmark score보다 failure mode coverage에 가깝다.

## Ultra: E-Commerce Packaging Deployment

Ultra section은 industrial robot이 existing workstation에 들어가 몇 시간 내 valuable work를 시작하는 방향을 설명한다. 첫 use case는 e-commerce order packaging이다. Article은 이 task가 item type, deformable packaging, external machinery, customer workflow variation 때문에 traditional automation으로는 long-tail이 크다고 설명한다.

Article이 공식적으로 적은 Ultra evidence는 다음과 같다.

| 항목 | 공식 글 기준 내용 | 해석 |
| --- | --- | --- |
| deployment context | live customer warehouses에서 real customer orders를 packing하는 videos라고 설명한다. | simulation이나 staged demo가 아니라 production-like setting을 강조한다. |
| full-shift evidence | π0.6가 Ultra customer site에서 full shift 동안 packaging orders를 수행했고 video caption은 96.4% autonomy라고 적는다. | article에서 가장 명시적인 Ultra 수치다. 다만 evaluation protocol 전체는 공개하지 않는다. |
| generation trend | π0 -> π0.5 -> π0.6마다 intelligence, throughput, reliability가 step up됐다고 설명한다. | family improvement를 partner deployment metric으로 연결한다. |
| success rate chart | π0.6 SFT가 π0.5 SFT보다 success rate가 높다고 chart/caption이 말한다. | model generation upgrade 효과다. |
| throughput chart | +UPT가 -UPT보다 order packing throughput을 높인다고 chart/caption이 말한다. | Ultra data pre-training이 throughput에 기여한다는 주장이다. |
| qualitative behavior | π0.6는 prompt adherence가 좋아지고 long-tail edge case에서 더 committed recovery strategy를 고른다고 설명한다. | success/failure binary 밖의 behavior quality를 강조한다. |
| human-in-the-loop | model이 stumble하면 human-in-the-loop intervention system이 correctness를 보장하고, thousands of orders에서 새 data를 만든다고 설명한다. | production reliability와 data flywheel이 연결된다. |

여기서 `UPT`는 Ultra data included in pre-training이다. `SFT`는 supervised fine-tuning이다. Ultra chart에는 95% confidence interval error bar가 표시되어 있다. 숫자를 정확히 재현하려면 원자료가 필요하지만, 공식 caption 기준으로는 “π0.6가 success rate를 올리고, +UPT가 throughput을 더 올린다”가 안전한 결론이다.

Ultra section의 tricky situation videos는 double pick 후 drop/regrab, 떨어진 item pick-up, bag machine 대기, funnel/bag unblock, funnel에 걸린 item recovery, 떨어진 polybag pick-up 같은 장면을 보여준다. 이들은 prompt adherence와 long-tail recovery claim의 qualitative evidence다.

## Model / Method Boundary

이 article은 모델 architecture, loss, training recipe를 공개하는 논문이 아니다. 따라서 “method”를 다음처럼 경계 지어야 한다.

| 물어볼 질문 | article에서 확인되는 것 | article만으로 확인되지 않는 것 |
| --- | --- | --- |
| 어떤 모델인가? | π0, π0.5, π0.6, π*0.6 계열 general-purpose robotic foundation model이라고 설명한다. | π0.6 partner checkpoint architecture/config, parameter count, action head detail |
| 어떤 학습인가? | SFT 조건과 WPT/UPT pre-training inclusion 조건이 chart label로 나온다. | data mixture 비율, optimizer, train step, augmentation, loss |
| 어떤 robot interface인가? | partners can interface their own hardware to PI models라고 설명한다. | camera/state/action schema, controller frequency, safety monitor |
| deployment 중 어떻게 개선되는가? | human-in-the-loop intervention이 correctness를 보장하고 새 data를 만든다고 설명한다. | intervention labels가 어떤 objective로 다시 학습되는지 |
| 공개 코드는 어디까지인가? | OpenPI는 π0/π0.5 계열의 research release와 fine-tuning path를 제공한다. | partner deployment stack, π0.6/π*0.6 production checkpoint 전체 |

따라서 이 문서는 implementation recipe가 아니라 platform/deployment evidence note로 분류한다. 모델 내부 구조를 보려면 [π0](../pi0-vla-flow-model/), [π0.5](../pi05-open-world-generalization/), [π*0.6 / RECAP](../pistar06-recap-learns-from-experience/) 쪽 개별 Survey를 읽어야 한다.

## PI Family 안에서의 위치

Physical Intelligence Layer article은 PI family를 “연구 결과들의 목록”이 아니라 “사용 가능한 robotics intelligence layer”로 재배치한다.

| 연구 축 | layer 관점에서의 의미 |
| --- | --- |
| [π0](../pi0-vla-flow-model/) | reusable low-level VLA foundation model의 시작점 |
| [OpenPI](../openpi-release/) | 공개 checkpoint, fine-tuning config, policy runtime으로 community가 실험할 수 있는 software layer |
| [π0-FAST](../pi0-fast-action-tokenization/) | action tokenization route로 language/action interface를 다르게 푸는 축 |
| [π0.5](../pi05-open-world-generalization/) | unseen environment와 open-world generalization을 강화한 family step |
| [Robot Olympics / π0.6](../moravec-robot-olympics/) | 새 physical skill acquisition과 fine-tuning capability를 보여주는 배경 |
| [π*0.6 / RECAP](../pistar06-recap-learns-from-experience/) | autonomous experience, correction, RL로 deployment reliability를 높이는 방향 |
| [RLT](../rlt-efficient-online-rl/) | online RL로 throughput/speed를 올리는 adaptation mechanism |
| [π0.7](../pi07-steerable-model/) | user steering, subgoal/current image conditioning으로 deployment controllability를 높이는 방향 |

이 article의 partner evidence는 위 논문들의 formal result를 대체하지 않는다. 대신 “이 family를 왜 product/platform layer로 포장하는가”를 설명하는 bridge 역할을 한다.

## 한계와 주의

- 이 문서는 논문이 아니라 partner/product article이다. formal benchmark protocol, dataset split, rollout count, random seed, baseline implementation detail이 공개되어 있지 않다.
- Weave/Ultra section은 partner가 작성한 deployment report 성격이므로 독립 재현 benchmark와 동일하게 해석하면 안 된다.
- Weave chart의 42% missed grasp 감소와 50% intervention 감소는 공식 caption의 claim이지만, full evaluation protocol은 article에서 확인할 수 없다.
- Ultra의 96.4% autonomy는 full-shift video caption에 나온 명시 수치다. 하지만 어떤 denominator와 intervention policy를 썼는지 세부 정의는 공개되지 않는다.
- `physical intelligence layer`는 단일 model checkpoint가 아니라 model, partner data, SFT, robot interface, runtime, intervention/data loop를 포함하는 platform abstraction이다.
- OpenPI처럼 공개된 부분과 partner deployment에서 쓰인 내부 stack을 구분해야 한다.
- public site에 공개하기 어려운 workplace, partner integration, production safety detail은 이 노트에 추정해서 넣지 않는다.
