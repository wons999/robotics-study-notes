## 한 줄 요약

OpenPI는 PI 계열 VLA를 논문 아이디어가 아니라 실제 checkpoint, data transform, fine-tuning config, policy wrapper, inference server로 만져볼 수 있게 공개한 software layer다.

## 공개 범위

공식 article은 OpenPI release의 목적을 “π0를 각자 robot/task에 fine-tune해 실험할 수 있게 하는 것”으로 설명한다. 공개 범위는 다음 네 가지다.

| 공개 항목 | 내용 |
| --- | --- |
| base model | `π0 base`, `π0-FAST base`, 이후 repo 기준 `π0.5 base` 계열 checkpoint |
| demo checkpoint | DROID, ALOHA, LIBERO 등 widely available platform용 fine-tuned checkpoint |
| inference example | real-world/simulated robot platform에서 policy를 실행하는 예시 |
| fine-tuning code | custom task/platform data로 base model을 fine-tune하는 code/config |

공식 article은 PI 내부 실험에서 1-20 hours data가 다양한 task fine-tuning에 충분했던 경우가 있다고 말하지만, 바로 caveat를 붙인다. π0는 원래 PI robot용으로 개발되었고, ALOHA/DROID 같은 public platform과 다르기 때문에 모든 robot에서 성공을 보장하지 않는다. 이 문장은 OpenPI를 “plug-and-play product”가 아니라 research release로 읽어야 한다는 근거다.

## 하드웨어 / 비용 감각

README는 OpenPI를 “가볍게 pip install 해서 노트북에서 돌리는 모델”로 설명하지 않는다. 최소 요구량은 사용 경로에 따라 크게 갈린다.

| 작업 | README 기준 메모리/장비 감각 | 해석 |
| --- | --- | --- |
| inference | 약 8 GB 이상 GPU memory | policy server를 GPU 머신에 두고 robot PC는 client로 분리하는 구성이 자연스럽다. |
| LoRA fine-tuning | 22.5 GB 이상, RTX 4090급 | custom task adaptation의 현실적인 entry point다. |
| full fine-tuning | 70 GB 이상, A100 80GB 또는 H100급 | 연구실 단위 compute assumption에 가깝다. |
| full DROID training | config comment 기준 약 8x H100, 100k steps, 약 2 days | OpenPI가 공개되어도 대규모 재학습은 여전히 비용이 크다. |
| full DROID data | README/예제 기준 DROID 1.0.1, 약 75k language annotations, 1.8TB storage | dataset 준비가 training 자체만큼 중요한 병목이다. |

DROID demo는 University of Montreal, University of Washington, KAIST 등 training에 없던 여러 location에서 실행된 예를 article이 든다. ALOHA demo도 training에 없던 새 ALOHA station에서 실행됐다고 설명한다. 다만 “new location에서 예시가 있다”와 “모든 robot/site에서 robust하다”는 다르다.

## Base / Expert Checkpoint 구분

OpenPI README의 checkpoint table은 `Use Case`를 함께 적는다. 이 구분이 중요하다.

| 구분 | 예시 | 어떻게 써야 하나 |
| --- | --- | --- |
| base model | `pi0_base`, `pi0_fast_base`, `pi05_base` | downstream dataset으로 fine-tune하는 출발점이다. |
| expert inference checkpoint | `pi0_fast_droid`, `pi0_aloha_towel`, `pi05_libero` | 특정 platform/task에서 바로 돌려보는 checkpoint다. 다만 setup mismatch에 민감하다. |
| inference/fine-tuning 겸용 | `pi05_droid` | DROID-like setup에서는 inference도 하고, smaller custom DROID data fine-tuning seed로도 쓴다. |

그래서 OpenPI 실험을 설계할 때는 “가장 최신 모델을 고른다”보다 “내 robot의 observation/action convention과 가장 가까운 checkpoint/config를 고른다”가 먼저다.

## Runtime Policy Contract

OpenPI를 실제로 실행할 때 핵심 단위는 “model class”가 아니라 `policy_config.create_trained_policy(config, checkpoint_dir)`가 만드는 policy object다.

| 단계 | 코드 기준 의미 |
| --- | --- |
| config 선택 | `_config.get_config("pi05_droid")`처럼 training/inference preset을 고른다. |
| checkpoint download | `download.maybe_download()`가 `gs://openpi-assets/...` checkpoint를 local cache로 가져온다. |
| backend detection | checkpoint directory 안에 `model.safetensors`가 있으면 PyTorch path, 없으면 JAX params path로 로드한다. |
| model load | JAX는 `params`를 `bfloat16`으로 restore하고, PyTorch는 safetensors를 load한 뒤 selected params를 bfloat16으로 맞춘다. |
| norm stats | checkpoint `assets` 아래 normalization statistics를 읽는다. robot/platform별 state/action scale이 여기서 맞춰진다. |
| input transforms | optional repack, default prompt injection, dataset-specific input mapping, normalization, model tokenization/image resize 순서로 적용된다. |
| output transforms | model output을 action transform, unnormalize, dataset-specific output mapping 순서로 robot action convention에 돌려놓는다. |
| policy output | `policy.infer(example)["actions"]`는 `[action_horizon, action_dim]` action chunk를 반환한다. |

remote inference 문서는 이 policy를 websocket server 뒤에 둔다. 서버는 기본 port `8000`에서 observation을 받고, robot/client 쪽은 image를 `224x224`로 resize/uint8 변환한 뒤 state와 prompt를 함께 보낸다. 문서가 강조하듯 state는 client에서 직접 normalize할 필요가 없고, normalization은 server-side policy transform이 처리한다.

### Remote Inference Contract

remote inference path는 다음 계약으로 이해하면 된다.

| 항목 | 내용 |
| --- | --- |
| server command | `uv run scripts/serve_policy.py policy:checkpoint --policy.config=... --policy.dir=...` |
| default port | `8000` |
| client package | `packages/openpi-client` |
| observation image | client에서 `224x224`, `uint8`로 resize/convert |
| observation state | unnormalized proprioceptive state를 보낼 수 있고, normalization은 server policy가 처리 |
| prompt | `prompt` key로 task instruction 전달 |
| output | `client.infer(observation)["actions"]` |
| output shape | `(action_horizon, action_dim)` action chunk |
| execution pattern | 매 timestep마다 호출하기보다 N step마다 action chunk를 받고 나머지는 open-loop 실행 |

이 구조는 robot software dependency와 GPU inference dependency를 분리한다. 실제 robot integration에서는 policy 자체보다 observation dictionary key, camera preprocessing, state/action convention을 맞추는 일이 더 자주 문제된다.

## Fine-Tuning Config Snapshot

`training/config.py`의 preset은 “OpenPI가 무엇을 쉽게 바꿔 보라고 설계했는가”를 보여준다.

| config | model config | data path | weight loader / steps |
| --- | --- | --- | --- |
| `pi0_libero` | `Pi0Config()` | `physical-intelligence/libero`, LeRobot | `pi0_base`, 30k steps |
| `pi0_fast_libero` | `Pi0FASTConfig(action_dim=7, action_horizon=10, max_token_len=180)` | `physical-intelligence/libero`, LeRobot | `pi0_fast_base`, 30k steps |
| `pi05_libero` | `Pi0Config(pi05=True, action_horizon=10, discrete_state_input=False)` | `physical-intelligence/libero`, LeRobot | `pi05_base`, batch 256, 30k steps |
| `pi0_aloha_pen_uncap` | `Pi0Config()` | `physical-intelligence/aloha_pen_uncap_diverse`, LeRobot | `pi0_base`, 20k steps |
| `pi05_aloha_pen_uncap` | `Pi0Config(pi05=True)` | same ALOHA dataset | `pi05_base`, batch 64, 20k steps |
| `pi0_fast_full_droid_finetune` | `Pi0FASTConfig(action_dim=8, action_horizon=16, max_token_len=180)` | full DROID RLDS | `pi0_fast_base`, batch 256, 100k steps |
| `pi05_full_droid_finetune` | `Pi0Config(pi05=True, action_dim=32, action_horizon=16)` | full DROID RLDS | `pi05_base`, batch 256, 100k steps |
| `pi05_droid_finetune` | `Pi0Config(pi05=True, action_dim=32, action_horizon=16)` | custom smaller DROID LeRobot dataset | `pi05_droid`, batch 32, 20k steps |

특히 DROID에는 두 경로가 있다. full DROID는 RLDS loader를 쓰고, README는 DROID `1.0.1`이 complete language annotations 약 75k episodes를 포함하며 1.8TB storage가 필요하다고 적는다. 반면 custom DROID fine-tuning은 비교적 작은 dataset을 LeRobot format으로 변환해서 돌리는 경로다.

## Fine-Tuning을 시작하기 전 체크리스트

OpenPI의 fine-tuning은 model weight보다 data contract가 먼저다.

| 체크 항목 | 왜 중요한가 |
| --- | --- |
| camera key | DROID/ALOHA/LIBERO마다 `observation/image`, wrist image 등 key가 다르다. |
| image preprocessing | policy server 예제는 `224x224` resize/uint8 conversion을 client에서 수행한다. |
| state dimension | `PadStatesAndActions(model_config.action_dim)`가 있으나, 의미 있는 state ordering은 사용자가 맞춰야 한다. |
| action dimension | `pi05_droid_finetune`은 `action_dim=32`, `pi0_fast_droid`는 `action_dim=8`처럼 config별로 다르다. |
| action horizon | LIBERO는 10, π0.5 DROID expert는 15, full DROID fine-tune은 16처럼 preset별로 다르다. |
| normalization stats | checkpoint `assets`에서 읽는 norm stats가 robot/data와 맞아야 한다. |
| prompt source | `prompt_from_task=True` 또는 default prompt injection이 model input을 바꾼다. |
| backend | JAX checkpoint인지 PyTorch `model.safetensors`인지에 따라 load path가 다르다. |

## Model / Transform Boundary

OpenPI에서 model config와 data transform은 분리되어 있다. 같은 `Pi0Config`라도 ALOHA/DROID/LIBERO에서 camera key, action convention, default prompt, delta action transform이 달라진다.

| 층 | 대표 코드 | 역할 |
| --- | --- | --- |
| model config | `Pi0Config`, `Pi0FASTConfig` | backbone/action head, action horizon, action dim, token budget 같은 model-side shape를 정한다. |
| data config | `LeRobotLiberoDataConfig`, `LeRobotAlohaDataConfig`, `RLDSDroidDataConfig` | dataset source, asset id, prompt source, robot-specific mapping을 정한다. |
| model transforms | `ModelTransformFactory` | π0/π0.5는 Paligemma prompt tokenization + image resize + state/action padding, π0-FAST는 FAST tokenization/extraction을 붙인다. |
| policy wrapper | `Policy` | transforms와 model sampling을 하나의 `infer()` API로 묶는다. |

이 구조 때문에 checkpoint 이름만 바꿔서 모든 robot에 바로 적용할 수는 없다. observation key, camera layout, state/action dimension, normalization asset, action horizon이 맞아야 한다.

### Policy Transform 순서

`policy_config.py` 기준 inference input은 다음 순서로 처리된다.

```text
raw robot observation
  -> optional repack_transforms.inputs
  -> InjectDefaultPrompt(default_prompt)
  -> data_config.data_transforms.inputs
  -> Normalize(norm_stats)
  -> data_config.model_transforms.inputs
  -> model.sample_actions(...)
  -> data_config.model_transforms.outputs
  -> Unnormalize(norm_stats)
  -> data_config.data_transforms.outputs
  -> optional repack_transforms.outputs
  -> robot action chunk
```

이 순서를 보면 OpenPI의 `Policy`는 단순 wrapper가 아니라, robot-specific schema와 model-specific tensor contract를 연결하는 adapter다. 따라서 OpenPI를 자기 robot에 붙일 때는 model architecture보다 transform chain을 먼저 읽어야 한다.

## Public Repo에서 보이는 모델 경계

OpenPI의 model boundary는 대략 다음과 같이 갈린다.

| 모델 | public repo에서 보이는 것 | public repo에서 보이지 않는 것 |
| --- | --- | --- |
| π0 | flow-matching model, base/expert checkpoint, fine-tuning config | PI 내부 full pretraining data recipe 전체 |
| π0-FAST | autoregressive token model, FAST tokenizer route, DROID/LIBERO/ALOHA style configs | PyTorch support, 최신 steering/runtime research |
| π0.5 | `pi05=True` config, base/DROID/LIBERO checkpoint, Knowledge Insulation 연결 | paper의 모든 hybrid co-training detail을 그대로 재현하는 end-to-end recipe |
| π*0.6 / RECAP | 관련 연구 문서와 비교 맥락 | public checkpoint/code inventory 없음 |
| RLT | 연구 문서와 π0.5/RL token 개념 | OpenPI main checkpoint inventory 없음 |
| π0.7 | 연구 문서에서 OpenPI 계열을 기반으로 설명 | public checkpoint/code inventory 없음 |

## PyTorch Support

현재 README는 PyTorch implementation을 제공한다고 설명한다. 단, 이것은 π0와 π0.5 중심이며 다음 caveat가 있다.

| 항목 | README 기준 상태 |
| --- | --- |
| 지원 모델 | π0, π0.5 |
| 미지원 | π0-FAST, FSDP training, LoRA training |
| checkpoint 변환 | JAX checkpoint를 PyTorch format으로 변환하는 script 제공 |
| inference API | `policy_config.create_trained_policy()` API는 JAX/PyTorch를 거의 동일하게 사용 |

따라서 OpenPI를 PyTorch-only stack으로 읽으면 안 된다. 특히 FAST route를 보려면 여전히 JAX 구현과 tokenizer path를 확인해야 한다.

추가로 주의할 점은 README의 “PyTorch에서 FSDP/LoRA 미지원”과 troubleshooting의 JAX/FSDP memory workaround를 섞지 않는 것이다. PyTorch path에서는 해당 기능이 제한되고, JAX training path에서는 별도 flag와 설정으로 multi-GPU memory trade-off를 다룬다.

## PI Family 안에서의 위치

OpenPI는 PI family의 실험 재현 축이다. `π0`는 foundation action model, `π0-FAST`는 token-action route, `π0.5`는 open-world generalization으로 이어지고, OpenPI는 이 모델들을 실제 robot data에 맞춰 fine-tune할 수 있는 entry point가 된다.

| 연결 문서 | OpenPI에서 확인할 것 |
| --- | --- |
| [π0](../pi0-vla-flow-model/) | flow matching action expert가 code에서 어떻게 구현되는지 |
| [π0-FAST](../pi0-fast-action-tokenization/) | FAST tokens가 raw model output이 아니라 transform을 거쳐 action으로 복원되는 구조 |
| [π0.5](../pi05-open-world-generalization/) | `pi05=True` config와 paper의 full hybrid co-training system 사이의 차이 |
| [The Physical Intelligence Layer](../physical-intelligence-layer/) | research code release와 partner/product layer가 어디서 갈라지는지 |

## 한계와 주의

- 공개 checkpoint는 baseline/fine-tuning seed로 봐야 하고, 각 robot의 calibration, camera pose, action convention, dataset schema가 맞지 않으면 실패할 수 있다.
- `π0-FAST base`는 training efficiency 장점과 runtime inference cost caveat를 같이 적어야 한다.
- demo checkpoint는 subsequent fine-tuning용이라기보다 out-of-box demonstration 성격이 강하다고 article이 설명한다.
- 최신 PI research article이 모두 OpenPI checkpoint로 공개된 것은 아니다.
- PyTorch support가 추가되었지만 π0-FAST/FSDP/LoRA training은 README 기준 아직 미지원이다.
- full DROID recipe는 1.8TB data, RLDS loader, 약 8x H100 규모의 compute assumption이 들어가므로 “가벼운 fine-tuning tutorial”과 같은 난이도로 보면 안 된다.
- remote inference에서 state normalization은 server-side policy가 처리하지만, state ordering과 action convention까지 자동으로 추론해 주지는 않는다.

## 스터디에서 기억할 점

OpenPI는 “논문 결과를 그대로 재현하는 버튼”이 아니라, PI 계열 모델을 내 robot/data에 연결하기 위한 공개 adapter stack이다. checkpoint는 model prior를 주고, config는 shape와 horizon을 정하고, transform은 robot schema를 model input/output으로 바꾸고, policy server는 GPU inference와 robot runtime을 분리한다. 이 네 층을 분리해서 읽으면 OpenPI를 과대평가하지 않으면서도 실제 실험 출발점으로 쓸 수 있다.
